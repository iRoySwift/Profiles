# 仓库协作说明

## 维护脚本

- 仓库维护任务优先使用 Rust，例如校验、排序、去重、冲突检查和批量更新。
- 除非 Rust 明显不合适，否则不要为这类维护任务临时使用 Ruby 或 Python。
- 维护脚本要保留在仓库里，方便后续重复使用。
- 维护脚本需要添加简洁注释，方便未来继续修改。
- 脚本位置优先放在稳定目录中，例如 `scripts/` 或 `tools/`。
- 当用户要求可复用的维护流程时，应实现为仓库内的 Rust 脚本，而不是一次性的临时命令。
- 如果任务属于同一维护领域，优先扩展现有 Rust 维护脚本，而不是再新建一套并行脚本。
- 脚本注释应保持“便于学习”的风格，方便仓库维护者后续自己阅读和学习。
- 对本仓库来说，注释应说明常量、结构体、函数以及重要分支或转换逻辑的用途。
- 如果某段逻辑后续大概率还会复用，就不要写成一次性的临时辅助脚本。
- 本仓库过滤器维护的统一入口脚本是 `scripts/filter_maintainer.rs`，优先扩展它，而不是再创建并行实现。
- Rust 维护脚本在任务完成后也要保留在仓库中，方便后续继续复用。
- 当维护逻辑发生变化时，要在同一轮同步更新 Rust 脚本、`README.md` 和 `AGENTS.md`。

## 过滤器文件处理

- 更新 `QuantumultX/Filter/*.list` 时，优先使用可复用的 Rust 脚本，而不是临时的 shell 处理命令。
- 排序、去重、头部标准化、冲突检查、宽泛规则清理，这些都视为同一类维护工作，并统一落到 Rust 脚本中。
- 域名类规则要按规则值的字母顺序排序；如果规则值相同，输出顺序也要保持稳定。
- 删除完全重复的规则。
- `.list` 文件的整理结果必须是幂等的，也就是脚本重复运行时不应反复产生无意义改动。
- 自动化修改后，文件头部统计必须和正文规则数量保持一致。
- 各个过滤器文件的头部格式要统一标准化。
- `UPDATED` 字段使用当前工作会话所在本地日期。
- 头部字段优先使用统一顺序；即使某种规则类型不存在，也要保留对应的 `0` 计数字段。
- 头部统计字段只保留当前 Quantumult X 实际使用的规则类型，不再单独展示 `DOMAIN` 系列计数头。
- 头部注释格式要统一，不要混用历史上不同风格。
- 清理冲突时，优先保留更具体的列表，并从更宽泛的列表中移除重复项或冲突项。
- 如果同一文件中、同一策略下，某条精确 `HOST` 或 `DOMAIN` 规则已经被现有的 `HOST-SUFFIX` 或 `DOMAIN-SUFFIX` 覆盖，则优先删除该精确规则，使列表更简洁。
- 进行规则简化时，只有在不会改变目标策略、也不会跨文件扩大匹配范围的前提下，才优先使用更宽泛但仍然精确安全的后缀规则。
- 如果多条精确主机规则可以明确、安全地合并为同策略的一条后缀规则，应优先使用后缀规则。
- 对明显过于宽泛、会导致过度匹配且没有必要保留的规则，应移除。
- `HOST-KEYWORD` 是合法规则，不能因为它是一段式关键字匹配就错误删除。
- 要保留合法的两段式规则，例如 `HOST-SUFFIX,example.com`，不能假设所有规则都一定带策略字段。
- 要保留合法的附加字段，例如 `IP-CIDR,1.2.3.4/32,no-resolve`。
- 排序和解析时，至少要支持以下规则类型：
  - `HOST`
  - `DOMAIN`
  - `HOST-KEYWORD`
  - `DOMAIN-KEYWORD`
  - `HOST-SUFFIX`
  - `DOMAIN-SUFFIX`
  - `HOST-WILDCARD`
  - `DOMAIN-WILDCARD`
  - `IP-CIDR`
  - `IP6-CIDR`
  - `IP-CIDR6`
  - `USER-AGENT`

## 冲突处理规则

- 冲突检查范围是整个 `QuantumultX/Filter/*.list`，而不是只看单个文件。
- 精确冲突检查应聚焦在带有明确策略目标的规则上。
- 不要把没有策略字段的两段式规则当成跨文件策略冲突。
- 当“同一规则类型 + 同一目标值”在不同文件中指向不同策略时，应视为冲突。
- 同文件内部的冗余检查要和跨文件冲突检查分开处理。
- 同文件冗余包括：某条精确 `HOST` 或 `DOMAIN` 规则已经被同策略的后缀规则覆盖。
- 自动解决冲突时，应尽量只做最小必要清理，避免额外改变语义。

## 头部标准

- 头部字段优先使用以下标准顺序：
  - `NAME`
  - `AUTHOR`
  - `REPO`
  - `UPDATED`
  - `HOST`
  - `HOST-KEYWORD`
  - `HOST-SUFFIX`
  - `HOST-WILDCARD`
  - `IP-CIDR`
  - `IP6-CIDR`
  - `IP-CIDR6`
  - `USER-AGENT`
  - `TOTAL`
- 缺失的计数字段不要省略，而是要补成 `0`。
- `TOTAL` 必须根据实际正文规则重新计算。

## 文档同步

- 只要新增或修改了仓库维护命令，就要在同一轮同步更新 `README.md`。
- README 中需要写明可直接运行的命令，以及它的简短用途说明。
- 如果 Rust 维护脚本新增了子命令，在 README 中应同时说明检查模式和写入模式（如果两者都存在）。
